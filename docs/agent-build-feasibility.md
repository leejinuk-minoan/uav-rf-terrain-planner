## 1. 결론 요약

본 프로젝트는 Codex 또는 Claude Code를 통해 상당 부분 자동화하여 빌드할 수 있다. 그러나 **사용자의 별도 통제 없이 처음부터 완성까지 전면 자율 빌드하는 것은 권장하지 않으며, 현실적으로도 완전 자동 완료를 보장하기 어렵다.**

가장 적절한 방식은 다음이다.

```text
사용자: 초기 방향 승인, 데이터 제공, 주요 단계 검토, PR merge 승인
AI 에이전트: 코드 작성, 테스트 작성, 문서화, 리팩터링, CI 구성, 이슈 단위 구현
```

즉, “무통제 완전 자동 빌드”가 아니라 **감독형 고자율 빌드**가 적절하다.

## 2. 판단 근거

### 2.1 프로젝트의 기술적 복잡도

본 프로젝트는 단순 웹앱이 아니라 다음 기능을 포함한다.

1. MGRS 좌표 변환
2. WGS84 및 미터 단위 투영좌표계 변환
3. DEM/DSM 래스터 처리
4. 지형 단면 추출
5. LOS 차폐분석
6. Fresnel Zone 침범률 계산
7. 발진기지 후보 격자 생성
8. 3D 운용반경 필터링
9. 경로탐색 알고리즘
10. 실 비행거리 계산
11. 약 500m 단위 경유점 산출
12. 지도 시각화 UI
13. 데이터셋 품질 검증
14. 테스트 및 시나리오 검증

이 중 코드 자체는 AI 에이전트가 상당 부분 작성할 수 있다. 그러나 지형 데이터 품질, 좌표계 선택, 분석 범위, 샘플링 해상도, 성능 최적화, 결과 신뢰성 검증은 사용자 또는 도메인 검토자가 반드시 확인해야 한다.

### 2.2 외부 데이터 의존성

프로그램은 DEM, DSM, 경사도, 토지피복도, 산림/임상도, 건물 높이/윤곽 데이터에 의존한다. 이 데이터들은 출처, 좌표계, 해상도, 라이선스, 파일 크기, 지역 범위가 다르다.

AI 에이전트가 샘플 데이터로 테스트 코드를 만들 수는 있지만, 실제 지역 데이터를 자동으로 확보하고 라이선스를 판단하며 정확한 좌표계로 전처리하는 작업은 완전 자동화 대상이 아니다.

### 2.3 모델 검증 필요성

LOS/Fresnel 기반 차폐점수는 물리적으로 타당한 1차 근사 모델이지만, 실제 링크 안정성을 완벽하게 보장하지 않는다. 따라서 다음 검증이 필요하다.

1. DEM 단면과 수작업 단면 비교
2. DSM 포함 여부에 따른 결과 변화 확인
3. 주파수 대역 변경에 따른 Fresnel 결과 변화 확인
4. AGL 고도 변화에 따른 추천 순위 변화 확인
5. 실제 또는 시뮬레이션 링크상태와 차폐점수 간 상관관계 확인
6. 실제 DEM/DSM 연결 후 직선 profile 기반 최소 요구 MSL 산출값 확인
7. 직선 운용구간 내 최고 지표고 기준 AGL 변환값 확인

이 검증은 AI 에이전트가 테스트 코드를 작성할 수 있지만, 검증 기준과 수용 여부는 사용자가 결정해야 한다.

최소 요구 MSL / AGL 산출은 실제 통신성공률 검증이 아니라 DSM 기반 LOS/Fresnel clearance proxy 검토로 정의한다. Android/TMMR offline은 제품화·배포 전략이므로 별도 feasibility 검토가 필요하다.

## 3. Codex 기반 빌드 가능성

### 3.1 가능한 작업

Codex는 다음 작업에 적합하다.

1. Python 프로젝트 스캐폴딩
2. 패키지 구조 설계
3. 좌표 변환 모듈 작성
4. DEM/DSM 래스터 샘플링 함수 작성
5. LOS/Fresnel 계산 함수 작성
6. A*/Dijkstra 경로탐색 구현
7. Streamlit/Folium 기반 MVP UI 작성
8. 단위 테스트 작성
9. GitHub Actions CI 작성
10. 문서화 및 README 정리
11. 이슈 단위 기능 구현
12. DSM 기반 LOS/Fresnel Clearance 조건을 만족하는 최소 요구 MSL 계산 scaffold 작성
13. 최고 지표고 기준 AGL 변환 helper 작성
12. PR 생성 및 반복 수정

### 3.2 한계

1. 사용자가 제공하지 않은 실제 지형 데이터를 임의로 확보할 수 없다.
2. 데이터 라이선스와 군사적·기관 내부 사용 가능 여부를 확정할 수 없다.
3. 결과 추천의 도메인 타당성을 독자적으로 보장할 수 없다.
4. GitHub 토큰, API 키, 외부 데이터 다운로드 권한은 사용자가 설정해야 한다.
5. 대형 GIS 데이터 처리 중 발생하는 성능 문제는 실제 환경 테스트가 필요하다.
6. 완성된 결과가 사용자의 의도와 일치하는지 최종 검수가 필요하다.
7. Android/TMMR offline 배포는 단말 성능, 저장공간, 오프라인 지도·지형 데이터 패키징, 현장 보안정책 검토가 필요하므로 별도 제품 로드맵으로 분리해야 한다.

### 3.3 적정 운용 방식

Codex는 “작은 단위의 구현 태스크”에 나누어 투입하는 것이 적합하다.

예시:

```text
Task 001: Python 패키지 구조와 pyproject.toml 생성
Task 002: MGRS/WGS84/투영좌표계 변환 모듈 구현
Task 003: DEM/DSM 래스터 샘플링 모듈 구현
Task 004: LOS 분석 함수와 테스트 작성
Task 005: Fresnel Zone 반경 및 침범률 계산 구현
Task 006: 발진기지 후보 격자 생성 및 점수화
Task 007: A* 경로탐색 기반 경로 후보 3개 생성
Task 008: 500m 단위 경유점 산출 및 표 생성
Task 009: Streamlit 지도 UI 구현
Task 010: CI, 테스트, 문서화 정리
```

## 4. Claude Code 기반 빌드 가능성

### 4.1 가능한 작업

Claude Code는 다음 작업에 적합하다.

1. 저장소 전체 구조 파악
2. 여러 파일에 걸친 기능 구현
3. 테스트 실행 및 실패 수정
4. 코드 리팩터링
5. Git commit 및 PR 생성
6. GitHub Actions 기반 자동화
7. CLAUDE.md 기반 프로젝트 규칙 준수
8. 반복 작업 자동화
9. 문서 업데이트
10. 리뷰 코멘트 반영

### 4.2 한계

1. 자동 모드라도 위험한 명령이나 외부 네트워크 접근은 승인/검토가 필요하다.
2. 데이터 다운로드, 대형 파일 처리, API 키 사용은 보안 관리가 필요하다.
3. 잘못된 지시, 문서 내 prompt injection, 악성 의존성에 취약할 수 있다.
4. 장시간 자율 작업은 비용, 토큰 사용량, GitHub Actions 시간 사용량을 증가시킨다.
5. 프로젝트 요구사항이 불명확하면 원하지 않는 기능이 구현될 수 있다.

### 4.3 적정 운용 방식

Claude Code는 다음 형태가 적절하다.

```text
1. CLAUDE.md에 프로젝트 규칙 명시
2. GitHub Issue 단위로 작업 부여
3. PR 생성 후 테스트 결과와 diff 검토
4. 사용자가 merge 승인
5. 다음 단계 이슈 진행
```

## 5. 무통제 완전 자동 빌드가 부적절한 이유

### 5.1 도메인 판단이 필요함

본 프로젝트는 “지도 위 경로가 그럴듯하게 보이는가”만으로 성공 여부를 판단할 수 없다. 좌표계, 지형고도, AGL/MSL 변환, 주파수 대역별 Fresnel 반경, DSM 장애물 반영 방식 등이 모두 결과에 영향을 준다.

### 5.2 데이터가 프로젝트 성능을 결정함

AI가 만든 코드가 맞더라도 DEM/DSM 데이터가 부정확하거나 해상도가 낮으면 결과가 틀릴 수 있다. 데이터 품질 검증은 자동화할 수 있지만, 수용 기준은 사용자가 정해야 한다.

### 5.3 보안 위험이 존재함

Codex와 Claude Code는 모두 파일 편집, 명령 실행, 테스트 실행 등 강력한 작업을 수행할 수 있다. 이 기능은 개발 속도를 높이지만, 의존성 설치, 스크립트 실행, 외부 네트워크 접근, 토큰 사용 시 보안 리스크가 발생한다.

### 5.4 완성 기준이 주관적임

“완성”은 단순히 코드가 실행되는 상태가 아니라 다음을 포함한다.

1. 사용자가 원하는 UI인가?
2. 추천 결과가 해석 가능한가?
3. 경로 후보 3개가 충분히 구분되는가?
4. 500m 경유점 표가 실무적으로 유용한가?
5. 속도가 운용 가능한 수준인가?
6. 테스트가 충분한가?
7. 문서와 실제 동작이 일치하는가?

이 항목은 사용자 검토 없이는 확정할 수 없다.

## 6. 권장 빌드 방식

### 6.1 감독형 고자율 빌드

본 프로젝트의 권장 방식은 다음과 같다.

```text
사용자 승인 필요:
- 데이터셋 제공/선택
- 주요 알고리즘 가정 승인
- MVP 범위 승인
- PR merge 승인
- 결과 화면 검수

AI 에이전트 자동화 가능:
- 코드 작성
- 테스트 작성
- 문서화
- 리팩터링
- CI 구성
- 샘플 데이터 기반 검증
- 이슈별 PR 생성
```

### 6.2 단계별 통제점

| 단계 | AI 자율작업 | 사용자 통제점 |
|---|---|---|
| Phase 0 | 저장소 구조화, 문서 정리 | 프로젝트 방향 승인 |
| Phase 1 | 좌표변환/데이터 로딩 구현 | 좌표계와 데이터 출처 승인 |
| Phase 2 | LOS/Fresnel 계산 구현 | 모델 산식 승인 |
| Phase 3 | 발진기지 추천 구현 | 점수 가중치 승인 |
| Phase 4 | 경로탐색 구현 | 경로 후보 방식 승인 |
| Phase 5 | UI 구현 | 화면 구성 승인 |
| Phase 6 | 테스트/CI | 검증 기준 승인 |
| Phase 7 | 릴리즈 | 최종 사용 승인 |

## 7. Codex와 Claude Code 역할 분담안

### 7.1 Codex 권장 역할

- 비동기 구현 작업
- 테스트 작성
- 반복적인 리팩터링
- 문서/코드 정합성 수정
- 작은 PR 단위 구현

### 7.2 Claude Code 권장 역할

- 로컬 환경에서 실행 확인
- 복잡한 디버깅
- 여러 파일을 동시에 이해해야 하는 리팩터링
- GitHub Actions 자동화
- 코드 리뷰 및 보안 점검

### 7.3 병행 사용 시 주의점

Codex와 Claude Code를 동시에 같은 브랜치에 투입하면 충돌 가능성이 높다. 반드시 기능별 브랜치를 분리해야 한다.

권장 브랜치 전략:

```text
main
├── agent/task-001-project-scaffold
├── agent/task-002-coordinate-core
├── agent/task-003-terrain-loader
├── agent/task-004-los-fresnel
├── agent/task-005-launch-recommendation
├── agent/task-006-route-planner
└── agent/task-007-streamlit-ui
```
