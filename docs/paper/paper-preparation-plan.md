## 5. 핵심 점수식 검증 계획

현재 점수식은 다음과 같다.

```text
발진 가능구역 종합점수 = 차폐안정성 점수 × 0.80 + 거리점수 × 0.20
차폐안정성 점수 = LOS 점수 × 0.50 + Fresnel 여유 점수 × 0.35 + DSM 장애물 점수 × 0.15
```

### 5.1 현재 판단

이 점수식은 MVP 초기 설계 가정으로 사용할 수 있다. 다만 논문에서는 검증된 최적 가중치가 아니라 **휴리스틱 가중치** 또는 **초기 설계 가중치**로 표현해야 한다.

### 5.2 타당한 근거

- LOS 차단은 통신 경로의 직접 차폐를 의미하므로 높은 비중을 부여할 수 있다.
- Fresnel Zone 침범은 LOS가 열려 있어도 신호 품질 저하 가능성을 설명하므로 독립 점수로 반영할 수 있다.
- DSM 장애물은 수목·건물 등 DEM만으로 반영하기 어려운 표면 객체 차폐를 보완한다.
- 거리점수는 운용반경 여유도를 반영하지만, 본 프로젝트의 핵심은 거리 최소화보다 차폐 안정성 평가이므로 낮은 비중으로 둘 수 있다.

### 5.3 한계

- 0.80/0.20, 0.50/0.35/0.15 가중치는 실제 링크상태 데이터로 보정된 값이 아니다.
- LOS와 DSM 장애물 점수는 독립이 아닐 수 있다. DSM 표면고도가 LOS/Fresnel 침범 계산에 이미 포함되면 중복 반영 위험이 있다.
- 거리점수는 운용반경 여유도에는 유용하지만, 링크 품질 자체를 직접 설명하지 않는다.
- 주파수 대역, 안테나 특성, 송신출력, 수신감도, 환경 잡음, 다중경로 등은 초기 점수식에 반영되지 않는다.

### 5.4 보완방안

| 보완안 | 내용 | 적용 단계 |
|---|---|---|
| A안: 현재 가중치 유지 | MVP에서는 초기 휴리스틱으로 유지하고 한계를 명시 | Task 007~008 |
| B안: 설정파일화 | 가중치를 config로 분리하여 실험별 조정 가능하게 설계 | Task 007 |
| C안: 민감도 분석 | 가중치 조합 변화에 따른 색상 분포와 경로 변화 비교 | Task 012~014 |
| D안: DSM 중복 방지 | DSM을 LOS/Fresnel 입력으로 쓸 때 DSM 장애물 점수 재정규화 | Task 007~008 |
| E안: 실제 링크 보정 | RSSI/SINR/packet loss 등 실제 링크 지표로 가중치 보정 | 향후 연구 |
| F안: radio map 확장 | terrain-derived shielding score를 실제 또는 ray-tracing radio map으로 대체/보정 | 향후 연구 |

### 5.5 논문 표현 기준

논문에는 다음처럼 표현한다.

```text
본 연구의 MVP 단계에서는 LOS 차단 여부, Fresnel Zone 여유도, DSM 기반 표면 장애물 침범 정도를 결합한 휴리스틱 차폐안정성 점수를 사용하였다. 해당 가중치는 실제 링크상태 데이터로 최적화된 값이 아니라, 지형 차폐 위험을 우선 반영하기 위한 초기 설계값이며, 향후 실제 DEM/DSM 및 링크상태 데이터 기반 민감도 분석과 보정이 필요하다.
```

최소 요구 고도 산출은 DSM 기반 LOS/Fresnel Clearance 조건을 만족하는 `minimum_required_msl_m`과 직선 운용구간 내 최고 지표고 기준 `required_agl_above_highest_dem_m`을 산출하는 방식으로 표현한다. 이 산출값은 실제 통신품질, 실제 정찰 성공, 실제 비행 가능을 보장하는 값이 아니라 공역사용승인 신청 고도 판단을 보조하기 위한 오프라인 모델 출력으로 기록한다.

## 8. 현재 결론

논문 작성 준비상태는 **부분 준비** 상태다.

- 프로젝트 목적, 기본 알고리즘, 안전 경계, 발진 가능구역 지도화 기준은 준비되어 있다.
- 논문 작성 책임을 GPT Master로 명확히 고정해야 한다.
- 점수식은 초기 설계 가정으로는 사용할 수 있으나, 실제 검증값으로 표현하면 안 된다.
- Task별 작업기록과 실험기록 구조를 먼저 만들고, 각 PR 종료 시 누적해야 한다.
- 실제 DEM/DSM 및 링크상태 데이터가 없으면 논문 결과는 synthetic 검증과 방법론 중심으로 제한해야 한다.
